# Artifacts Generation Summary

This document describes how epics, features, and user stories are generated by the Workflow system: the APIs involved, the AI prompt & settings, the server processing steps, frontend integration, schemas, and debugging tips.

## High-level flow

- User drives a conversation in the Workflow UI (Step 1).
- When the assistant has enough information, the frontend calls `/api/workflow/generate-guidelines` (optional) and then `/api/workflow/generate-artifacts`.
- The server assembles a compact system prompt + the requirement text and calls the AI (Azure/OpenAI).
- The AI returns structured JSON containing `epics`, `features`, `userStories`, and `personas`.
- The server parses and validates the JSON, returns it to the frontend, and saves artifacts to an SDLC project via `/api/workflow/save-artifacts`.
- The frontend stores artifacts in `WorkflowProvider` (context) using `setEpics`, `setFeatures`, `setUserStories` and renders them.

## Key API endpoints

- `POST /api/workflow/conversation`
  - Purpose: Ask the AI for the next question or determine readiness to generate artifacts.
  - Request body (example):
    - `conversationHistory`: array of `{ role, content }`
    - `capturedRequirements`: object with arrays for businessGoals, keyFeatures, etc.
    - `currentPhase`, `askedQuestions`, `isRegenerating`, `originalRequirement`
  - Response (example): `{ question, phase, quickReplies?, singleSelect?, capturedInfo?, readyToGenerate? }`

- `POST /api/workflow/generate-guidelines`
  - Purpose: (Optional) generate AI design guidelines from the requirement text.
  - Request body: `{ input: string }` where `input` is assembled requirement text.
  - Response: `{ guidelines: string }`

- `POST /api/workflow/generate-artifacts`
  - Purpose: Produce the main artifacts (epics/features/userStories/personas).
  - Request body (example):
    - `requirement`: full requirement text (conversation + capturedRequirements)
    - `complianceGuidelines`: server-side guidelines, if any
    - `selectedPersonaIds`: array of persona ids to bias generation
  - Response (example):
    {
      epics: [{ id, title, description, acceptanceCriteria? }],
      features: [{ id, epicId, title, description, acceptanceCriteria? }],
      userStories: [{ id, epicId, featureId, title, description, acceptanceCriteria?, tasks? }],
      personas: [{ id, name, summary }]
    }
  - Notes: The server expects valid JSON. If the AI returns malformed JSON this endpoint will return an error with parse diagnostics.

- `POST /api/workflow/save-artifacts`
  - Purpose: Persist generated artifacts into the SDLC/project system.
  - Request body: `{ sessionId, projectId, requirement, epics, features, userStories, personas }`
  - Response: project save result (depends on SDLC API).

- `POST /api/sdlc/projects`
  - Purpose: Create an SDLC project if none exists before saving artifacts.
  - Request body: `{ name, description, status }`.

## AI call & prompt details

- Model settings (observed in current codebase):
  - Temperature: low (deterministic behavior expected)
  - `max_tokens`: increased during debugging to allow large JSON (example final value used: `8000`)
  - Instruction: compact system prompt + user requirement content

- Prompt strategy:
  - The server constructs `requirementText` which includes a conversation transcript and a normalized list of captured requirements (business goals, target users, key features, constraints).
  - A concise system prompt instructs the model to output structured JSON with keys `epics`, `features`, `userStories`, `personas` and to keep the output strictly valid JSON.
  - To reduce risk of truncation and token exhaustion the system prompt is kept short and focused.

- Example prompt summary (abstracted):
  - System: You are an agile backlog generator. Given a requirement, return JSON with arrays `epics`, `features`, `userStories`, `personas`. Each item must include `id` and `title`. Keep output strictly valid JSON only.
  - User: [requirementText]

## Server-side processing (step-by-step)

1. Receive request to `/api/workflow/generate-artifacts` with `requirement` and other options.
2. Assemble the prompt: compact system prompt + requirement text. Ensure the prompt is as compact as possible to maximize response tokens.
3. Call the AI service (Azure/OpenAI client) with low temperature and high `max_tokens` (to allow large JSON responses).
4. Receive streaming or full response. If streaming, buffer until complete. The server logs the full response length for diagnostics.
5. Attempt to parse the response as JSON. If parsing fails, the server logs the raw response, the response length, and the parse error (e.g., "Unterminated string in JSON at position 21292"). The server returns a descriptive error to the caller.
6. Validate JSON schema:
   - Ensure `epics`, `features`, `userStories` are arrays.
   - Each `feature` should reference `epicId` that exists in `epics` (best-effort).
   - Each `userStory` should reference `featureId` and `epicId`.
   - Ensure required fields like `id`, `title` are present.
7. If validation passes, return the JSON to the frontend and proceed to save:
   - If no SDLC project id is present, create a project via `POST /api/sdlc/projects`.
   - Call `/api/workflow/save-artifacts` to persist artifacts.
8. If saving fails, return success for generation (and error for save) so the user can still see artifacts and retry save.

## Frontend integration (how UI uses artifacts)

- Context: `client/src/context/workflow-context.tsx` provides `epics`, `features`, `userStories`, and setters `setEpics`, `setFeatures`, `setUserStories` plus selection toggles (`toggleEpic`, `toggleFeature`, `toggleStory`).
- Flow in Step 1 (`step1-conversational-refinement.tsx`):
  1. User interacts with the assistant. When ready, a dialog lets the user choose to `Generate Artifacts`.
  2. The frontend calls `/api/workflow/generate-guidelines` (optional) and `/api/workflow/generate-artifacts`.
  3. On success, the frontend executes `setEpics(...)`, `setFeatures(...)`, `setUserStories(...)`, and `setPersonas(...)` with the returned payload.
  4. The UI displays artifacts in the right context panel. The new hierarchical dropdown behavior maps as:
     - Epics dropdown: lists `epics` by `title`.
     - Features dropdown: filters `features` where `feature.epicId === selectedEpicId`.
     - User Stories list: filters `userStories` where `story.featureId === selectedFeatureId` and shows story summary/points. Clicking a story opens a details dialog showing description, acceptance criteria, and subtasks.

## JSON schema (expected shape)

- Epic
  - `id` (string)
  - `title` (string)
  - `description` (string, optional)
  - `acceptanceCriteria` (array|string, optional)

- Feature
  - `id` (string)
  - `epicId` (string)
  - `title` (string)
  - `description` (string, optional)

- UserStory
  - `id` (string)
  - `epicId` (string)
  - `featureId` (string)
  - `title` (string)
  - `description` (string, optional)
  - `acceptanceCriteria` (array|string, optional)
  - `tasks` (array|string, optional)

## Testing and debug tips

- Test script: `test-artifact-generation.js` (a smoke test exists in the repo) — it calls the generate endpoint and validates returned counts and presence of ids.
- Common failure: JSON parse errors caused by token truncation. Signs:
  - Server log: `Unterminated string in JSON at position N` and a short length value.
  - Fixes:
    - Increase `max_tokens` in the AI call.
    - Shorten or condense system prompt and examples.
    - Prefer a single concise system instruction and push more content into the user `requirement` payload.
    - If streaming, ensure the server buffers until EOS before parsing.
- Logging recommendations:
  - Log response length and the first/last ~200 characters of AI output when parsing fails.
  - Persist failing raw responses (redacted) for root-cause analysis.

## Error handling & UX

- If artifact generation fails (parse error or AI error), the frontend shows a descriptive toast with server diagnostic text.
- If saving to SDLC fails after successful generation, show a warning toast but keep artifacts in context so the user can retry saving.

## Security & operational notes

- Reuse a single AI client instance, enable retries for transient network errors, and respect rate limits.
- Do not log full user secrets; redact PII from stored failing responses if persisted.

## Next steps & improvements

- Consider emitting partial artifacts progressively (e.g., epics first, then features, then stories) to reduce single-response size.
- Add JSON-schema validation and a small auto-repair step (e.g., try to wrap truncated JSON or attempt to fix common trailing commas).
- Add telemetry for artifact sizes that trigger token exhaustion so you can tune `max_tokens` per tenant.
